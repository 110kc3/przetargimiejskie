## 2. Automation — the refresh now runs DAILY (was weekly)

**The gap:** `refresh.yml` was scheduled `cron: '0 6 * * 1'` — **Mondays only**.
The inline comment even said "Daily also fine — runs are short," but it was never
switched. So the BIP crawl ran once a week.

**Fixed:** changed to `cron: '0 4 * * *'` → **04:00 UTC daily ≈ 06:00 Polish
time** (summer/CEST). GitHub cron is UTC-only and cannot follow DST, so in winter
(CET) it fires 05:00 local — that is the one unavoidable caveat.

**The rest of the chain was already automated** (verified, no change needed):

| Stage | Workflow | Trigger | Status |
|---|---|---|---|
| Crawl BIPs → commit `data/<city>/*.json` | refresh.yml | daily cron (now) | ✅ |
| Rebuild `data/index.json` | refresh.yml `index` job | after all city jobs | ✅ |
| Redeploy live site (OVH) | ovh-deploy.yml | `workflow_run` after refresh | ✅ |
| Freshness guard | health.yml | 07:00 UTC daily | ✅ |
| Extension data | reads raw.githubusercontent directly | n/a | ✅ always fresh |

Note: the deploy correctly uses a `workflow_run` trigger because the bot commits
data with `GITHUB_TOKEN`, and GitHub deliberately does not let a `GITHUB_TOKEN`
push trigger downstream workflows. So switching the cron to daily is genuinely the
only change required for end-to-end daily automation.

Optional follow-up (not done, to avoid surprising failures): now the crawl is
daily, `STALE_DAYS` in health.yml could be tightened from 14 → ~3 to catch a
stalled city within days instead of two weeks.

## 4. Google-Maps links — how they're populated (and they're fine)

Flats/houses/commercial/garages get a Maps link built **client-side in the
extension** (`popup.js` / `archive.js`, function `mapsCell`):

```
base = street + " " + building   (else address_raw / addr_display, apt number stripped)
href = https://www.google.com/maps/search/?api=1&query=<base>, <city>
```

Land plots (`kind: 'grunt'`) deliberately get **no** Maps link — they use the
parcel/geoportal column instead.

**Verified:** 0 of the non-land properties across all cities are missing a Maps
base — every flat/house resolves to a usable Maps query. So the Maps column is
healthy.

One latent inconsistency (no current impact, reported not fixed): `popup.js`
builds the base from `address_raw` only, while `archive.js` falls back to
`street + building`. Today 0 active listings are affected (all have
`address_raw`), but if an active listing ever lacks `address_raw` while having
`street`, the popup would show "—" where the archive would show a link. Aligning
the two `mapsCell` functions would be a one-line hardening (extension change →
needs a version bump, so left for you to decide).

## 5. Geoportal links — how they're populated, and the real gaps

Land plots resolve a map link in `pipeline/src/core/geoportal.js`, in priority
order:

1. a link the **source** embedded (e.g. Bytom's SIP) — used verbatim;
2. a **per-city override** (municipal SIP / e-mapa);
3. a **precise national-geoportal deep-link** (`mapy.geoportal.gov.pl/imap/?identifyParcel=<TERYT id>`)
   when a full TERYT parcel id is known — resolved from obręb+parcel via ULDK
   (`uldk.gugik.gov.pl`) in the `enrich-land` step, cached in `uldk-cache/`;
4. a **Google-search fallback** when no precise id can be built.

**Coverage across 477 land plots:**

| Link type | Count | |
|---|---|---|
| Precise geoportal deep-link (identifyParcel) | 216 | good |
| Source/municipal link | 12 | good |
| Weak Google-search fallback | **249** | the problem |
| Null (no link at all) | 0 | |

The **249 weak links** split into two very different cases:

- **221 are upgradeable** — they HAVE a parcel number; ULDK just hasn't resolved
  them to a TERYT id. **Sosnowiec alone is 164 of these.** Diagnosis: ULDK *was*
  run (cache exists) and returned **NULL for 163/226 Sosnowiec parcels** because
  the obręb is passed malformed — e.g. `"11 Sosnowiec"`, `"0012 Sosnowiec"`
  (inconsistent 2-digit vs 4-digit, and polluted with the city name). ULDK can't
  match that. **Fix:** normalise Sosnowiec's obręb (strip the city name, use a
  consistent obręb code) before the ULDK call, then re-run `enrich-land`.
  **FIXED & APPLIED (25 June):** added a *construct-and-verify* path to
  `uldk.js` — when the obręb carries a number, build the full id
  `gmina.OOOO.parcel` (obręb code zero-padded) and confirm it via ULDK
  `GetParcelById`, bypassing the unmatchable placeholder name. Re-ran
  `enrich-land` for Sosnowiec: **precise links 63 → 157, weak 164 → 70**
  (+94 recovered). The 70 still-weak are parcels that genuinely don't exist
  in the cadastre under that obręb/number — correctly left as a fallback,
  never a wrong link. New unit tests in `tests/uldk.test.js`.
- **28 are addr-only** — no parcel number at all, so a precise parcel deep-link is
  *impossible*. These are inherently fallback-only (Mysłowice 8, Tarnowskie Góry
  12, Świętochłowice 5, Katowice 1, Zabrze 2).

### Fix applied (safe, test-verified)

The fallback for addr-only plots was throwing the street away: it built the
search from the **obręb code only**, ignoring the street the plot actually has.

Before: `działka 0005 Krasowy Mysłowice geoportal działki`
After:  `działka Orła Białego 0005 Krasowy Mysłowice geoportal działki`

`geoportal.js` now includes `plot.street` (falling back to `address_raw`) in the
fallback query, so addr-only plots produce a meaningful search instead of a
near-useless obręb-code one. Added a regression test
(`tests/geoportal.test.js`); the geoportal + land-builder tests pass. The
improved links flow into committed data on the next daily refresh.
